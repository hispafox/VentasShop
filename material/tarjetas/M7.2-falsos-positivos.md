# Tarjeta · Falsos positivos y tests frágiles

> Un test solo vale si su resultado significa algo. Dos formas de verde que no protege: **aserción débil**
> (pasa siempre) y **test inestable** (pasa a veces). La herramienta que las caza: **mutation testing**.

## Aserción débil vs fuerte

```csharp
Assert.True(tasa >= 0);      // ❌ se cumple para 0%, 15%, 99%
Assert.True(tasa <= 0.15m);  // ❌ se cumple para 0%, 5%, 10%
Assert.Null(ex);             // ❌ solo "no petó"
Assert.Equal(0.15m, tasa);   // ✅ el valor exacto: la única que caza un bug de cálculo
```

Regla: **afirma el resultado concreto que esperas, no una propiedad vaga.**

## Mutation testing (Stryker.NET, sobre MTP)

```bash
dotnet tool install -g dotnet-stryker
cd tests/VentasShop.TestsUnitarios
dotnet stryker --test-runner mtp     # preview; el repo va sobre MTP, sin VSTest
```

- La herramienta planta bugs (`0.10m`→`0.0m`, `>=`→`>`); cada uno es un **mutante**.
- **killed**: un test falló (bien). **survived**: ningún test se enteró (mal).
- **mutation score** = % de mutantes matados = calidad de tus tests (no cobertura).
- Es **lento**: de vez en cuando sobre la lógica crítica, no en cada commit.
- NO Coverlet, NO Docker.

## Tests inestables (flaky) — 4 causas y cura

- **Tiempo** (la más común): `DateTimeOffset.Now` → fija «ahora» con `RelojFijo` (IReloj, M5).
- **Estado compartido**: pasa/falla según el orden → independencia (FIRST, M1.3).
- **Concurrencia / async** mal esperado → esperar de verdad, no asumir orden.
- **Dependencias externas** (red, disco, BD) en un unitario → dobles (M5).
- Matiz: a veces destapa una *race condition* real → investiga antes de borrar.

## Errores

- Reintentar hasta que pase (esconde). · Perseguir 100% de mutation (mutantes equivalentes).
- Confundir cobertura con mutation score. · Matar al mensajero (borrar el test / debilitar la aserción).
