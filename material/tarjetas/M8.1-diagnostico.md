# Tarjeta · Diagnóstico del proyecto

> Un proyecto real no se parece a VentasShop: sin tests, acoplado, da miedo tocarlo. El primer trabajo no es
> testear, es **diagnosticar** y abrir la primera costura. Abre el Módulo 8. (Legacy NEUTRO inventado, R1.)

## El orden (urgencias y triaje)

1. **Triaje** — clasificar antes de curar. Riesgo × consecuencia (M2.1) + **historial de Git** (qué se toca
   mucho). Empieza por la cocina, no por el recibidor.
2. **Diagnóstico de testabilidad** — busca las **costuras** (seam). Las dependencias clavadas lo impiden.
3. **Intervención mínima** — abre una costura con DI, **sin cambiar el comportamiento**.
4. Si da respeto, primero **test de caracterización**, luego el refactor.

## Los clavos que impiden el test (legacy)

```csharp
new RepositorioPedidosSql("Server=prod-db;...")   // acceso a datos clavado
DateTime.Now                                       // reloj del sistema (no determinista)
new SmtpClient(...) ; smtp.Send(...)               // manda correo de verdad
Logger.Instance.Log(...)                           // singleton estático global
```

Sin sitio para meter un doble → no hay test unitario (M5).

## Costura (seam) — Feathers

Un sitio donde cambias el comportamiento sin editar ese código (sustituyendo una dependencia). VentasShop
está lleno (cada interfaz inyectada); el legacy, soldado. Si la tiene, testeas hoy; si no, ábrela.

## La intervención mínima (DI al revés)

Los 4 clavos → 4 parámetros del constructor: `IAccesoPedidos`, `IReloj`, `IEnviadorCorreo`, `ILogger<>`. La
lógica idéntica línea a línea. **Cambias la estructura, no el comportamiento.** Ahora se le pasan dobles.

## El huevo y la gallina

- Refactors que abren costuras = mecánicos y de bajísimo riesgo (los hace el IDE) → antes de tener tests.
- **Test de caracterización**: fija lo que el código *hace ahora* (no lo correcto); red para refactorizar.

## Errores

- Empezar por donde no toca · querer dejarlo bonito · refactor sin red en código crítico · diagnosticar para siempre.
