# Tarjeta · Testing con EF Core in-memory

> Tarjeta de 1 página. El in-memory es un **simulador de conducción**: rápido y cómodo para practicar la
> lógica, pero no es la carretera (un motor relacional). El equipo de EF lo desaconseja para lo relacional.

## Cómo se usa

```csharp
private static ContextoVentasShop CrearContexto(string nombreBd) =>
    new(new DbContextOptionsBuilder<ContextoVentasShop>()
        .UseInMemoryDatabase(nombreBd).Options);
```

- Paquete `Microsoft.EntityFrameworkCore.InMemory`.
- **Nombre único por test** (`Guid`) = independencia. **Mismo nombre** en dos contextos = mismo almacén.
- **Lee en un contexto NUEVO**, no en el que guardaste (si no, EF te da la caché de seguimiento → falso verde).
- **`Include(p => p.Lineas)`** para cargar navegaciones: `Find` por sí solo no las trae.

## Qué NO comprueba (la letra pequeña)

- **Índice único** (guarda dos `Producto.Codigo` iguales sin rechistar).
- **Integridad referencial de FK** y `CHECK`.
- **Traducción a SQL** (ejecuta en memoria; una consulta que el motor rechazaría, aquí pasa).
- **Transacciones reales, índices, triggers.**
- **Matiz (EF Core 6+):** los `NOT NULL`/required **SÍ** los valida. Lo demás de esta lista, no.

## El ejemplo que lo demuestra

```csharp
contexto.Productos.Add(new Producto { Codigo = "ABC", Nombre = "Teclado", PrecioUnitario = new Dinero(50m, "EUR") });
contexto.Productos.Add(new Producto { Codigo = "ABC", Nombre = "Ratón",   PrecioUnitario = new Dinero(20m, "EUR") });

var excepcion = Record.Exception(() => contexto.SaveChanges());
excepcion.Should().BeNull();   // in-memory NO valida el índice único; SQLite in-memory (M6.3) SÍ lanzaría
```

## Cuándo sí

- Probar la **lógica** de un repositorio o consulta donde las restricciones no son lo que verificas.
- Tests rápidos sin instalar nada, sabiendo que no validas el comportamiento relacional.
- Siguiente escalón: **SQLite in-memory** (motor relacional real, sin instalar nada, pero no es tu motor exacto), en M6.3.

## Errores comunes

- Confiar en él para **validar restricciones** (no lo hace).
- **Leer del mismo contexto** con el que escribiste (caché de EF).
- **Compartir el nombre** de la base entre tests (se pisan).
- Creer que tienes tests de tu **capa de datos** cuando solo tienes tests de su **lógica**.
