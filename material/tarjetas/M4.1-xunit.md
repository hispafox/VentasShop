# Tarjeta · Introducción a xUnit.net

> Tarjeta de 1 página. Montar bien el proyecto de tests es lo que hace que ejecutar la suite sea un gesto
> de un segundo. Y un gesto de un segundo se repite; un ritual de varios pasos, no.

## Montar el proyecto (xUnit v3)

```bash
dotnet new install xunit.v3.templates                                   # plantillas (1 vez por maquina)
dotnet new xunit3 -o tests/VentasShop.TestsUnitarios                    # crear el proyecto de test
dotnet add tests/VentasShop.TestsUnitarios reference src/VentasShop.Dominio   # la flecha de dependencia
```

- Producción y tests en **proyectos separados**, misma solución. La dependencia va en un solo sentido:
  el test conoce a producción, **nunca** al revés.
- El `.csproj` de v3 trae `<OutputType>Exe</OutputType>` (el test es ejecutable) y el paquete
  `xunit.v3.mtp-v2` sobre la **Microsoft Testing Platform**. Por debajo cambia; `dotnet test` sigue igual.

## Un test es un `[Fact]`

```csharp
[Fact]
public void CalcularTasaDescuento_PedidoDe500ClienteEstandar_Aplica10PorCiento()
{
    var calculadora = new CalculadoraDescuentos();          // Arrange
    decimal tasa = calculadora.CalcularTasaDescuento(500m, TipoCliente.Estandar);  // Act
    Assert.Equal(0.10m, tasa);                              // Assert (esperado primero)
}
```

`[Fact]` = un hecho que siempre debe ser cierto. La clase agrupa los tests del SUT (M3.1), el nombre se
lee (M3.2), el cuerpo es AAA (M3.1).

## El repertorio de `Assert` (los más usados)

| Comprobación | Aserción |
|---|---|
| Igualdad (el rey) | `Assert.Equal(esperado, real)` |
| Condición booleana | `Assert.True` / `Assert.False` |
| Nulo / no nulo | `Assert.Null` / `Assert.NotNull` |
| Colección | `Assert.Contains` · `Assert.Empty` · `Assert.Single` |
| Tipo | `Assert.IsType<T>(objeto)` |
| Excepción | `Assert.Throws<T>(() => ...)` *(a fondo en M4.3)* |

- **Usa la aserción más específica que encaje:** `Assert.Empty(lista)` dice "esperaba vacía, tenía 3";
  `Assert.True(lista.Count == 0)` solo dice "esperaba True". El mensaje de fallo es el premio.
- **Orden de `Assert.Equal`: esperado primero, real después.** Si lo inviertes, el mensaje sale del revés.

## Ejecutar y leer el fallo

```bash
dotnet test                 # toda la suite (lo que usa la CI, M7)
```

- En el editor, el **Test Explorer**: ejecutar uno/todos, verdes y rojos de un vistazo, breakpoint + depurar.
- ¿Cada cuánto? **Constantemente** (FIRST, M1.3): como los unitarios son rápidos, los ejecutas tras cada cambio.
- Un test rojo te da nombre + esperado vs obtenido + dónde. Léelo de arriba abajo: te lleva al sitio.

## Malentendidos del primer día

- No compila / no ve producción → falta la referencia, o mezclas paquetes de v2 y v3.
- Tutorial de v2 en proyecto v3 → lo que escribes es igual; cambia el montaje.
- `Assert.Equal` al revés → si el mensaje suena "del revés", revisa el orden.
- No ejecutar a menudo → ejecuta sobre la marcha, no veinte tests de golpe al final.
