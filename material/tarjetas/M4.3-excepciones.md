# Tarjeta · Gestión de excepciones en tests

> Tarjeta de 1 página. Que el código falle bien —la excepción correcta, el mensaje correcto— es una
> funcionalidad, y se testea. Se prueba el airbag provocando el choque a propósito.

## `Assert.Throws<T>` (tipo EXACTO)

```csharp
var pedido = new PedidoBuilder().SinLineas().Build();
Assert.Throws<PedidoSinLineasException>(() => pedido.Pagar());
```

- La acción va en una **lambda dentro** del `Throws` (xUnit la ejecuta con su try/catch). Si la llamas
  fuera, la excepción tumba el test antes de la aserción.
- Exige el **tipo exacto**: esperar la clase padre cuando se lanza la hija → falla. Te obliga a ser preciso.
- Para aceptar el tipo **y sus subtipos**: `Assert.ThrowsAny<T>` (a conciencia, no por pereza).

## Verificar mensaje y propiedades

`Assert.Throws<T>` **devuelve la excepción**: la capturas y la interrogas.

```csharp
var ex = Assert.Throws<ArgumentException>(() => new Cantidad(0));
Assert.Equal("valor", ex.ParamName);              // propiedad = contrato estable
Assert.Contains("mayor que cero", ex.Message);    // FRAGMENTO, no el texto exacto
```

- Mensaje con `Contains`, **no** con `Equal`: si afirmas el texto entero, una tilde corregida rompe el
  test sin que haya bug (falsa alarma). Prioriza propiedades estructuradas sobre el texto.

## Async: `Assert.ThrowsAsync<T>` (OJO al `await`)

```csharp
await Assert.ThrowsAsync<PagoRechazadoException>(() => servicio.PagarAsync(idPedido));
```

- **Sin el `await`, el test pasa SIEMPRE** aunque el código no lance nada: el `Task` se descarta, la
  aserción no se evalúa. Falso positivo silencioso, el error más caro del módulo.
- Regla: `await` siempre + el método de test es **`async Task`**, nunca `async void`.
- Analizador que lo caza: **xUnit2021** (+ xUnit1031). Tenlo activado.

## `Record.Exception`: comprobar que NO lanza

```csharp
var excepcion = Record.Exception(() => pedido.Pagar());
Assert.Null(excepcion);   // un pago válido no dispara ningún airbag
```

## Solo se testean las excepciones de DOMINIO

- **Sí:** las que tu código lanza a propósito como contrato — `PedidoSinLineasException`,
  `TransicionPedidoInvalidaException`, la `ArgumentException` de una invariante.
- **No:** las de un bug interno (`NullReferenceException` por no validar) — eso se arregla, no se testea.
- La pregunta correcta: no "¿qué excepciones pueden salir?", sino "¿qué excepciones prometo lanzar?".
