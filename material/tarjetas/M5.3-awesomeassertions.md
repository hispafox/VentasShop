# Tarjeta · Aserciones fluidas con AwesomeAssertions

> Tarjeta de 1 página. Una aserción fluida no solo se lee mejor: cuando falla, te da el número, no un
> "anormal". La ganancia está en el mensaje de fallo, multiplicado por cada fallo del proyecto.

## El repertorio

```csharp
tasa.Should().Be(0.10m);                       // valor exacto (no BeGreaterThan(0))
pedido.Estado.Should().Be(EstadoPedido.Pagado);
pedido.Cliente.Nombre.Should().StartWith("Cliente");
pedido.Lineas.Should().HaveCount(3);
pedido.Lineas.Should().NotBeEmpty();
pedido.Lineas.Should().Contain(l => l.Cantidad.Valor > 1);
pedido.Lineas.Should().AllSatisfy(l => l.Subtotal.Importe.Should().BePositive());
```

Las de colección ahorran un bucle en el test (lógica en el test = antipatrón M3.1) y, al fallar, nombran el
elemento que no cumplía.

## La joya: `BeEquivalentTo`

```csharp
real.Should().BeEquivalentTo(esperado);
// Id autogenerados (Guid) → exclúyelos, o el test falla por ellos:
real.Should().BeEquivalentTo(esperado, opc => opc.Excluding(m => m.Name == "Id"));
```

Compara el objeto entero por valor, recorriendo el grafo. Brilla en integración (M6). Para una sola
propiedad, comprueba esa propiedad.

## Excepciones fluidas

```csharp
pedido.Invoking(p => p.Pagar())
      .Should().Throw<PedidoSinLineasException>()
      .WithMessage("*sin líneas*");   // fragmento con comodín, no texto exacto
```

(Forma alternativa: `Action acto = () => pedido.Pagar(); acto.Should().Throw<...>()`.)

## El porqué en el mensaje: `.Because()`

```csharp
pedido.Estado.Should().Be(EstadoPedido.Pagado,
    "porque la pasarela aceptó el pago y el flujo debe marcarlo así");
```

Aparece en el fallo. Úsalo donde el porqué no sea obvio, no en cada línea.

## Por qué AwesomeAssertions

FluentAssertions v8 (enero 2025) pasó a licencia de pago. AwesomeAssertions = fork libre de la v7, **API
idéntico**; solo cambia el paquete y el namespace (`using AwesomeAssertions;`). Shouldly es la otra gratuita.

## Errores comunes

- **Débiles más bonitas** — `BeGreaterThan(0)` comprueba casi nada; usa el valor exacto.
- **Abusar de `BeEquivalentTo`** — compara el objeto entero solo cuando de verdad lo necesitas.
- **Encadenar de más** — si no cabe de un vistazo, parte en varias aserciones.
- **Mezclar estilos** — elige nativo o fluido para el proyecto y sé consistente.
