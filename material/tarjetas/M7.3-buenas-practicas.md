# Tarjeta · Buenas prácticas consolidadas

> El test es código de producción: lo lees más veces de las que lo escribes. Una suite se cuida como un
> **jardín** (se poda, se limpia) o se vuelve una **jungla** que nadie toca. Cierra el Módulo 7.

## Un assert conceptual por test

- Un comportamiento, una razón para el rojo. El `FuncionaTodo` con 4 aserciones de 4 cosas = anti-ejemplo.
- La primera aserción que falla **corta el test** (las siguientes ni se ejecutan).
- «Conceptual» ≠ una línea `Assert`: varias propiedades del **mismo** resultado van juntas. Cuentas
  comportamientos, no líneas.

## Simplicidad

- Ni `if`/`for`/cálculos en el cuerpo del test (¿quién testea al test?).
- Varios casos → `[Theory]` con `[InlineData]` (M4.2), no un `for`.
- El número «mágico» con significado en un test es **bueno**: `Assert.Equal(0.15m, tasa)` se entiende solo.

## Independencia y determinismo (FIRST, M1.3)

- Nunca estado mutable compartido · nunca asumir orden · nunca reloj/red/fichero.
- Cada test fabrica sus datos con el `PedidoBuilder` (M3.3). Si el orden cambia algo → problema de diseño.

## Refactor: DAMP, no DRY

- Producción: **DRY** (no te repitas). Tests: **DAMP** (*Descriptive And Meaningful Phrases*).
- Quita el **ruido** (montaje farragoso → builder); conserva la **historia** (qué datos y qué afirma, a la vista).
- **Borra sin culpa** los tests de funcionalidades muertas (Git se acuerda; la suite no es museo).

## Tests como red de CI/CD

- El **gate**: la suite se ejecuta sola en cada cambio y bloquea lo que rompe.
- Rápidos (un gate lento se esquiva) y deterministas (uno que falla al azar se desactiva). Montaje = M8.2.

## Errores comunes

- Write-once (no volver a tocar la suite) · test comentado/saltado eterno · sobre-DRY (ilegible) · el
  `FuncionaTodo` que lo comprueba todo.
