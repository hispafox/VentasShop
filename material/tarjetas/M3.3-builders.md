# Tarjeta · Fixtures, Builders y Object Mother

> Tarjeta de 1 página. Si el 80% de un test es preparar datos, no estás probando: estás amueblando. El
> atrezzo de test devuelve el foco a lo que de verdad se comprueba.

## La regla de fondo

En el Arrange, **lo relevante para el test se ve; el andamiaje se esconde.** Si pruebas el descuento de
un VIP, el "VIP" a la vista; que el cliente se llame Ana, no.

## Los tres patrones

| Patrón | Qué es | Cuándo |
|---|---|---|
| **Object Mother** | Métodos estáticos que devuelven arquetipos ya montados (`ClienteMother.Vip()`) | Pocos arquetipos fijos que repites tal cual |
| **Test Data Builder** | Objeto que encadena decisiones y construye con `Build()` | Variaciones a medida (combinatoria) |
| **Fixture** | Recurso montado una vez para varios tests (`IClassFixture<T>`) | Algo caro de crear que NO cambia |

Conviven: un Object Mother puede usar por dentro un Builder con valores por defecto.

```csharp
var pedido = new PedidoBuilder().ParaVip().ConLinea(300m, 2).Pagado().Build();
// "un pedido de un VIP, con una linea de 300 EUR por 2 unidades, ya pagado"
```

## xUnit: setup, teardown e independencia

- **Constructor = setup** (antes de cada test). **`IDisposable.Dispose` = teardown** (después). C# puro.
- **Una instancia nueva de la clase de test por cada test:** cero estado arrastrado. Te da la
  independencia de FIRST (M1.3) por defecto.
- **`IClassFixture<T>`** comparte un recurso entre los tests de la clase. Útil para lo caro (una conexión).

## El peligro (el error más caro)

- **Estado compartido mutable:** si compartes algo que un test modifica, el siguiente lo hereda sucio →
  tests que fallan según el orden.
- **Regla:** comparte lo caro que **no cambia**; nunca estado que los tests modifican. Lo barato, créalo
  por test. (Los fixtures de BD, a fondo en **M6**.)
