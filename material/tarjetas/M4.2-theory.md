# Tarjeta · Tests parametrizados con `[Theory]`

> Tarjeta de 1 página. Un mismo comportamiento con muchos datos no son muchos tests: es un test con
> muchos casos. Parametrizar es separar el molde (la lógica) de las láminas (los datos).

## `[Theory]` + `[InlineData]` (constantes simples)

```csharp
[Theory]
[InlineData(1)]
[InlineData(5)]
[InlineData(100)]
public void Cantidad_ConValorPositivo_GuardaElValor(int valor)
    => Assert.Equal(valor, new Cantidad(valor).Valor);
```

- `[Theory]` = test que se ejecuta varias veces; cada `[InlineData]` = un juego de datos.
- **Cada `[InlineData]` es un test independiente:** una fila por caso en el Test Explorer. Un `foreach`
  dentro de un `[Fact]` se pararía en el primer fallo y te ocultaría el resto.
- `[InlineData]` solo admite **constantes de compilación**: enteros, cadenas, booleanos, enums.

## La limitación que importa en este curso

`decimal` **no** es una constante de atributo válida en C#: `[InlineData(0.10m, ...)]` **no compila**.
Como los importes son `decimal`, van por las otras fuentes.

## Las otras tres fuentes

| Fuente | Qué es | Cuándo |
|---|---|---|
| `[MemberData]` | Casos de un método/propiedad estático | Decimales, objetos, casos generados o compartidos |
| `TheoryData<>` | `[MemberData]` **tipado** (chequeo en compilación) | La opción por defecto para importes/objetos |
| `[ClassData]` | Una clase que hereda de `TheoryData<>` | Datos compartidos entre clases o con su propio archivo |

```csharp
public static TheoryData<decimal, TipoCliente, decimal> Casos => new()
{
    { 50m,  TipoCliente.Estandar, 0.00m },
    { 800m, TipoCliente.Vip,      0.15m },   // tope
};

[Theory]
[MemberData(nameof(Casos))]   // usa nameof, no la cadena: refactor-safe
public void CalcularTasaDescuento(decimal total, TipoCliente tipo, decimal esperado)
    => Assert.Equal(esperado, new CalculadoraDescuentos().CalcularTasaDescuento(total, tipo));
```

## Cuándo parametrizar y cuándo NO

- **Parametriza** cuando es el mismo comportamiento con datos distintos.
- **Separa** cuando son comportamientos distintos.
- El olfato: ¿solo cambian los valores, o cambia lo que el test verifica?
- **El antipatrón:** un `bool debeLanzar` + `if` dentro del `[Theory]` = dos tests en uno (calcular y
  rechazar entrada inválida). Si aparece un `if` para diferenciar casos, son dos tests. Sepáralos.

## Errores comunes

- `[Theory]` con un solo `[InlineData]` → usa `[Fact]`.
- `object[]` de ocho columnas sin nombres → `TheoryData<>` tipado o comenta las columnas.
- Parametrizar de más hasta esconder la intención → la claridad manda sobre el ahorro de líneas.
