# Tarjeta · Definir la estrategia

> Un proyecto real está en producción y **no se puede parar**. La estrategia es un plan por fases, no
> llenar de tests a lo loco. El entusiasmo no es una estrategia. (Reparar el barco sin dejar de navegar.)

## El plan por fases (de lo que más hunde a lo que más pule)

- **F1 — línea de flotación**: unitarios del **núcleo de negocio crítico**. Mayor retorno; usa las costuras
  de M8.1 + dobles (M5). Empieza aquí.
- **F2 — maniobra**: **integración** en las fronteras (BD, servicios externos). Más lenta; va después. Hay
  fallos que solo caza el motor real (M6.3).
- **F3 — automatismo**: el **gate de CI**. Se monta aquí (M7.3 solo lo nombró).

## El gate de CI (F3)

```bash
dotnet restore ; dotnet build ; dotnet test   # si falla -> rojo -> no entra
```

- GitHub Actions / Azure Pipelines (sin fijar versión de YAML). Ver `material/ci-ejemplo.yml`.
- Cobertura en la misma pasada: `dotnet test -- --coverage` (Microsoft Code Coverage/MTP). **NO Coverlet.**
- Reparto: unitarios en cada push · integración en cada merge · mutation testing (M7.2) nocturno.

## Objetivo de cobertura (que no sea mentira)

- **NO** 80% global con fecha (desmoraliza o invita a la trampa de M2.3).
- **SÍ** por zona (crítico de F1 alto, 80-90% ramas; andamiaje bajo) + creciente: **cobertura sobre código
  nuevo** (M7.1). La global sube sola.

## Confirmar el stack

Con el equipo, una vez, **por escrito antes de empezar**. Cambiarlo a mitad = carísimo.

## Errores

- Querer parar el barco · empezar por F2/F3 · objetivo global con fecha · planificar para siempre (el plan
  cabe en una página y se empieza).
