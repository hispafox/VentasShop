# Tarjeta · ¿Automatizar este test o dejarlo a mano?

> Tarjeta de decisión de 1 página. Tenla al lado cuando decidas qué merece un test automatizado.

## El filtro (tres preguntas)

1. **¿La vas a repetir muchas veces?** Una sola vez → a mano.
2. **¿El resultado es siempre el mismo (determinista)?** Depende del criterio → a mano.
3. **¿Comprobarla a mano es lento o aburrido?** Trivial → a mano. Tedioso → **automatízala**.

Regla en una línea: **automatiza lo repetitivo y determinista; deja el criterio y la exploración para
las personas.**

## Cada uno en lo suyo

| | Bueno para | Malo para |
|---|---|---|
| **Automatizado** | reglas de negocio, comprobaciones que se repiten en cada release, bordes (0, null, límites) | explorar algo nuevo, juzgar si "se ve bien" |
| **Manual / exploratorio** | pantallas nuevas, corazonadas ("¿y si meto un emoji?"), criterio visual | repetir las mismas 40 comprobaciones cada release (te aburres y fallas) |

## Para qué sirve un test (de menos a más importante)

1. Detectar errores (lo que todos dicen; lo menos valioso).
2. Documentar qué hace el código — y **fallar en rojo** si deja de cumplirlo.
3. **Dejarte tocar el código sin miedo** (el corazón: refactorizar con red).

## Las dos trampas

- **Frecuencia oculta:** "una vez" hoy puede ser "cada despliegue" mañana → entonces se automatiza.
- **Dos comprobaciones en una:** "el PDF queda bien" = *se genera* (automatizable) + *queda bonito*
  (criterio). Sepáralas.

> La primera habilidad del testing no es teclear `[Fact]`: es decidir **qué merece automatizarse**.
