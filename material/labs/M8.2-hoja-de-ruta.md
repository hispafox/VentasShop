# Lab M8.2 · La hoja de ruta en una página

> Submódulo M8.2. Tiempo: ~40-50 min. Es de **estratega, no de programador**: no escribes tests, diseñas
> el plan que los ordena. Partes del legacy diagnosticado en M8.1 (ya con sus costuras abiertas) y lo
> conviertes en una hoja de ruta de testing en **una página** que el equipo pueda ejecutar el lunes.

## El encargo

Para el mini-proyecto legacy de M8.1, redacta su estrategia por fases. La entrega no es una suite de tests:
es el plan. Cabe en una página y tiene cuatro bloques.

### 1. Fase F1 — la línea de flotación (unitarios del núcleo)

Lista las **2-3 clases críticas** (lógica de negocio: cálculo, reglas, transiciones de estado) que entran
primero, **con su justificación de riesgo** (si esto se rompe, ¿qué pasa? ¿se toca mucho?). Son las que
blindas con unitarios rápidos, apoyándote en las costuras que abriste en M8.1 + dobles (M5).

### 2. Fase F2 — los sistemas de maniobra (integración en fronteras)

Señala las **fronteras** que van a integración: el acceso a base de datos, las llamadas a servicios
externos. Recuerda que hay fallos que solo se cazan contra un motor relacional real (SQLite in-memory, M6.3).

### 3. Fase F3 — el gate de CI

Esboza el pipeline (ver [`material/ci-ejemplo.yml`](../ci-ejemplo.yml)): restaurar → compilar → `dotnet test`
que bloquea el merge si falla. Decide el **reparto**: qué se ejecuta en cada push (los unitarios rápidos),
qué en cada merge (la integración) y qué de noche (el mutation testing de M7.2, lo pesado). Cobertura en la
misma pasada con `dotnet test -- --coverage` (Microsoft Code Coverage; **no** Coverlet).

### 4. Objetivo de cobertura por zona

Nada de «80% global para fin de trimestre». Pon un objetivo **por zona** (la lógica crítica de F1 alto,
cerca del 80-90% de ramas; el andamiaje, bajo) y **creciente** (cobertura sobre código nuevo, M7.1).

## Ejemplo de plan resuelto (esquema)

```
LEGACY X — Hoja de ruta de testing
F1 (unitarios):   NotificadorPedidos (notifica segun hora -> riesgo medio, se toca a menudo)
                  [+ la clase de calculo critico que tenga el legacy]  -> objetivo 85% ramas
F2 (integracion): frontera con BD (acceso a pedidos) ; envio de correo (servicio externo, doblar)
F3 (gate CI):     push -> unitarios ; merge -> + integracion ; nocturno -> mutation testing
Cobertura:        F1 critico 85% ramas ; resto: cobertura sobre codigo nuevo (no bajar)
Stack:            confirmado con el equipo (xUnit v3 + NSubstitute + AwesomeAssertions + MS Code Coverage/MTP)
```

## Comprobación

No hay tests que ejecutar en este lab (es estrategia). La suite del repo sigue verde:

```bash

```
