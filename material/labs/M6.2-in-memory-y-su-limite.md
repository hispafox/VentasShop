# Lab M6.2 · EF Core in-memory y su límite

> Submódulo M6.2. Tiempo: ~35-45 min. Escribes tus primeros tests de integración con el provider in-memory
> de EF Core —lo cómodo— y, sobre todo, ves con tus ojos qué **no** comprueba —lo revelador—. Estrena el
> proyecto `VentasShop.TestsIntegracion`.

## Lo que vas a conseguir

Practicar la lógica de tu repositorio contra «una base de datos» rápida y sin instalar nada, y comprobar de primera
mano que el in-memory no refuerza un índice único. Es el simulador de conducción: vale para el trazado, no
es la carretera.

## Punto de partida

El modelo resuelto está en
[`tests/.../RepositorioPedidosInMemoryTests.cs`](../../tests/VentasShop.TestsIntegracion/RepositorioPedidosInMemoryTests.cs).
`Microsoft.EntityFrameworkCore.InMemory` y `AwesomeAssertions` ya están en el `.csproj`. `Producto` estrena
el campo `Codigo` con índice único en
[`ContextoVentasShop`](../../src/VentasShop.Infraestructura/Persistencia/ContextoVentasShop.cs).

## Parte 1 — Montar el contexto in-memory

Escribe el helper que crea el contexto:

```csharp
private static ContextoVentasShop CrearContexto(string nombreBd) =>
    new(new DbContextOptionsBuilder<ContextoVentasShop>()
        .UseInMemoryDatabase(nombreBd)
        .Options);
```

Recuerda: nombre único por test (`Guid`) = independencia; mismo nombre en dos contextos = mismo almacén.

## Parte 2 — Guardar y recuperar (lo cómodo)

Guarda un pedido con `new RepositorioPedidos(contexto).Agregar(pedido)` (que guarda dentro) y recupéralo en
un **contexto nuevo** (misma base) con `Include(p => p.Lineas)`. Comprueba que no es nulo y que trae su línea.

**Detalle clave:** lee en contexto nuevo, no en el que guardaste. Si lees del mismo, EF te devuelve el objeto
de su caché de seguimiento y el test pasa sin tocar la base. Y `Find` no carga navegaciones: por eso el
`Include`.

## Parte 3 — Una consulta (sigue cómodo)

Escribe el test de «pedidos de un cliente»: guarda dos pedidos de Ana y uno de Beto, y verifica que la
consulta `lectura.Pedidos.Include(p => p.Cliente).Where(p => p.Cliente.Id == ana.Id)` devuelve 2. Esto es
exactamente para lo que el in-memory es bueno: probar la **lógica** de tus consultas, rápido.

## Parte 4 — El límite (lo revelador)

Aquí está el momento del módulo. Añade dos productos con el **mismo** `Codigo` y guarda:

```csharp
contexto.Productos.Add(new Producto { Codigo = "ABC", Nombre = "Teclado", PrecioUnitario = new Dinero(50m, "EUR"), UnidadesStock = 10 });
contexto.Productos.Add(new Producto { Codigo = "ABC", Nombre = "Ratón",   PrecioUnitario = new Dinero(20m, "EUR"), UnidadesStock = 10 });

var excepcion = Record.Exception(() => contexto.SaveChanges());
excepcion.Should().BeNull();   // el in-memory NO valida el índice único
```

El test **pasa** porque el in-memory no refuerza el índice único: guarda los dos sin lanzar. Esa es la
limitación, documentada en verde. Si esperaras `Throw<DbUpdateException>()` (como contra una BD real), el
test fallaría — justo el aviso del módulo.

## Para comprobar

- [ ] Usas un nombre de base único por test, y el mismo nombre para guardar y leer.
- [ ] Lees en un contexto nuevo y con `Include` para traer las líneas.
- [ ] Has visto que `SaveChanges` con código duplicado NO lanza en in-memory (`Record.Exception` → null).
- [ ] Entiendes que contra un motor real (M6.3) ese mismo test sí lanzaría.

## Lo que viene

En M6.3 nos subimos al circuito con un coche de verdad: SQLite in-memory, un motor relacional real, sin instalar nada.
Volverás a este mismo test de la unicidad y, esta vez, **sí** saltará la `DbUpdateException`. Ahí verás la
diferencia entre el simulador y la carretera.
