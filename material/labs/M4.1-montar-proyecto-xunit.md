# Lab M4.1 · Montar el proyecto de tests y escribir tu primer `[Fact]`

> Submódulo M4.1 (Tests Unitarios con xUnit.net). Tiempo: ~30-40 min. Es la primera vez que montas
> testing con tus manos: del proyecto vacío a una suite en verde, y de ahí a leer un fallo de verdad.

## Lo que vas a conseguir

Arrancar el proyecto de tests de VentasShop como si no existiera, escribir tus primeros `[Fact]` sobre
la `CalculadoraDescuentos`, ejecutarlos hasta verlos verdes y, después, romper el código a propósito
para aprender a leer la salida de un fallo. Poco código; mucho banco de trabajo.

## Punto de partida

En esta rama el proyecto `tests/VentasShop.TestsUnitarios` ya está montado (lo arrastras desde el
Módulo 3). Para el lab, **haz como si lo montaras de cero**: los pasos de abajo son los que seguirías en
un proyecto nuevo, y el resultado tiene que quedar igual que lo que ya ves en el repo. Si te animas,
créalo en una carpeta aparte y compáralo con este.

## Parte 1 — Montar el banco

1. **Instala las plantillas de xUnit v3** (una vez por máquina):

   ```bash
   dotnet new install xunit.v3.templates
   ```

2. **Genera el proyecto de tests** apuntando a la carpeta de tests:

   ```bash
   dotnet new xunit3 -o tests/VentasShop.TestsUnitarios
   ```

3. **Traza la flecha de dependencia** (el proyecto de test conoce al de producción, nunca al revés):

   ```bash
   dotnet add tests/VentasShop.TestsUnitarios reference src/VentasShop.Dominio
   ```

4. **Abre el `.csproj` generado** y fíjate en dos cosas, que son la novedad de la v3:
   - `<OutputType>Exe</OutputType>` — el proyecto de test es un ejecutable, no una librería.
   - el paquete es `xunit.v3.mtp-v2` con `<TestingPlatformDotnetTestSupport>true</...>` — la variante
     que se apoya en la **Microsoft Testing Platform**. (En el repo lo tienes tal cual.)

## Parte 2 — Tu primer `[Fact]`

Crea una clase de tests para la calculadora y escribe el primer test. Tienes el modelo en
[`tests/.../PrimerasPruebasXunitTests.cs`](../../tests/VentasShop.TestsUnitarios/PrimerasPruebasXunitTests.cs):

```csharp
[Fact]
public void CalcularTasaDescuento_PedidoDe500ClienteEstandar_Aplica10PorCiento()
{
    // Arrange
    var calculadora = new CalculadoraDescuentos();

    // Act
    decimal tasa = calculadora.CalcularTasaDescuento(500m, TipoCliente.Estandar);

    // Assert
    Assert.Equal(0.10m, tasa);   // esperado primero, real despues
}
```

Reconoces todo de los módulos anteriores: la clase agrupa los tests del SUT (M3.1), el nombre se lee
como una frase (M3.2), el cuerpo sigue AAA (M3.1). Lo único nuevo es el `[Fact]` y el `Assert.Equal`.

Ahora **añade dos `[Fact]` más**, uno para el tramo medio (un pedido de 200 €, que da el 5%) y otro para
el tramo bajo (un pedido de 50 €, que no da descuento). Sí: vas a copiar y pegar casi lo mismo cambiando
dos números. Aguanta esa incomodidad, que la usamos en M4.2.

## Parte 3 — Ejecutar y ver verde

```bash
dotnet test tests/VentasShop.TestsUnitarios
```

Lee el resumen: cuántos pasaron, cuánto tardó. Abre también el **Test Explorer** de tu editor y ejecútalos
desde ahí; ese panel es tu compañero del día a día.

## Parte 4 — Romper a propósito y leer el fallo

Aquí está la parte que de verdad enseña. Abre
[`src/VentasShop.Dominio/Precios/CalculadoraDescuentos.cs`](../../src/VentasShop.Dominio/Precios/CalculadoraDescuentos.cs)
y cambia el `>= 500m` por `> 500m` (el bug clásico del límite). Vuelve a ejecutar.

Tu test del tramo alto se pone rojo. Lee la salida de arriba abajo:

- el **nombre** del test te dice qué comportamiento se rompió,
- el **mensaje** te dice qué esperaba (`0.10`) frente a qué obtuvo (`0.05`),
- con eso ya sabes que el descuento se quedó en el tramo medio: el `500` justo dejó de entrar en el alto.

Deshaz el cambio y comprueba que todo vuelve a verde.

## Para comprobar

- [ ] El proyecto compila y los `[Fact]` aparecen en el Test Explorer.
- [ ] Tus tres tests (alto, medio, bajo) pasan en verde.
- [ ] Al meter el bug del `>`, cae **solo** el test del tramo alto, y el mensaje te dice por qué.
- [ ] Sabes decir, sin mirar la solución, por qué el orden de `Assert.Equal` es esperado-primero.

## Pista para lo que viene

Para cubrir bien la calculadora vas a necesitar muchos más casos —tramos, tipos de cliente, límites, el
tope del 15%—, y escribirlos como `[Fact]` separados se vuelve insostenible. Esa es la puerta a los tests
parametrizados, `[Theory]`, que es M4.2. Compara tu fichero de `[Fact]` con el
[`CalculadoraDescuentosTests.cs`](../../tests/VentasShop.TestsUnitarios/CalculadoraDescuentosTests.cs)
ya parametrizado: es el mismo trabajo, sin la repetición.
