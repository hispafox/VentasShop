# Lab M5.1 · Elegir el doble de cada dependencia (lab de criterio)

> Submódulo M5.1 (Mocking, Aislamiento y Aserciones Fluidas). Tiempo: ~25-30 min. **Es un lab de
> criterio, no de código todavía:** decides qué doble necesita cada dependencia y qué vas a verificar.
> Con esa decisión clara, escribir el código en M5.2 con NSubstitute es casi mecánico. Abre el Módulo 5.

## Lo que vas a conseguir

Mirar el `ServicioPedidos` —que orquesta el pago de un pedido con cuatro dependencias— y, para cada
escenario de test, decidir con criterio el tipo de doble de cada dependencia (dummy, stub, spy, mock o
fake) y si vas a verificar **estado** o **interacción**. Es el mismo ejercicio de la servilleta de M2.1,
ahora aplicado a los dobles.

## El SUT y sus dependencias

```csharp
public class ServicioPedidos(
    IRepositorioPedidos repositorio,   // lee y guarda pedidos
    IPasarelaPago pasarelaPago,        // cobra de verdad, a un servicio externo
    IReloj reloj,                      // da la fecha y hora actuales
    ILogger<ServicioPedidos> registro) // registra eventos de auditoría
{
    public void Pagar(Guid idPedido) { /* obtiene, cobra, paga, guarda, registra */ }
}
```

Recuerda las reglas de negocio en juego: si la pasarela rechaza, el pedido **no avanza ni se guarda**
(BR-09); al pagar se deja un apunte de auditoría (BR-10); la fecha la fija el reloj para que el test sea
repetible (BR-06).

## Parte 2 — ¿Estado o interacción?

Para cada escenario, decide qué vas a comprobar al final del test:

- **Verificación de estado** — miras un resultado (el `Estado` del pedido quedó en `Pagado`).
- **Verificación de interacción** — compruebas que el SUT *llamó* a una dependencia (se intentó cobrar,
  se guardó, se registró).

Anota cuál usas en cada escenario y recuerda la regla: **prefiere estado siempre que puedas**; reserva
la interacción para cuando el comportamiento que importa *es* la llamada en sí (que se cobró, que se
auditó), porque ahí no hay un estado propio que mirar.
