# Lab M3.2 — Renombrar tests para que cuenten la noticia

> En este lab no escribes comportamiento nuevo: coges una batería de tests con nombres malos y los
> renombras para que, al fallar, te digan qué se rompió sin abrir el código. El formato:
> `Metodo_Escenario_ResultadoEsperado`.

## El punto de partida

Imagina que heredas estos cinco tests. Funcionan, pasan en verde, y sus nombres no valen nada. Lee el
**cuerpo** de cada uno (no el nombre) para saber qué prueba de verdad.

```csharp
// 1)
[Fact]
public void Test1()
{
    var calc = new CalculadoraDescuentos();
    Assert.Equal(0.10m, calc.CalcularTasaDescuento(500m, TipoCliente.Estandar));
}

// 2)
[Fact]
public void TestDescuento2()
{
    var calc = new CalculadoraDescuentos();
    Assert.Equal(0.15m, calc.CalcularTasaDescuento(800m, TipoCliente.Vip));   // 10% + 5% = tope 15%
}

// 3)  el nombre describe la implementación, no el comportamiento
[Fact]
public void CalcularTasaDescuento_UsaSwitch_DevuelveDecimal()
{
    var calc = new CalculadoraDescuentos();
    Assert.Equal(0.05m, calc.CalcularTasaDescuento(200m, TipoCliente.Estandar));
}

// 4)  el nombre MIENTE: dice "pagado" pero comprueba el envío
[Fact]
public void Pagar_PedidoConfirmado_PasaAPagado_OtroMas()
{
    var pedido = PedidoPagado();
    pedido.Enviar();
    Assert.Equal(EstadoPedido.Enviado, pedido.Estado);
}

// 5)
[Fact]
public void TestExcepcion()
{
    var pedido = new Pedido(new Cliente { Nombre = "Ana" });   // sin líneas
    Assert.Throws<PedidoSinLineasException>(() => pedido.Pagar());
}
```

## Parte 1 — Diagnostica

Sin renombrar todavía, responde:

1. Imagina los cinco en el panel del pipeline, tres en rojo. ¿Sabrías qué se rompió? ¿Cuál de los cinco
   nombres es el **más peligroso** y por qué? (Pista: no es el más vago.)
2. El test 3 pasa hoy. Si mañana cambias el `switch` de `CalcularTasaDescuento` por un diccionario sin
   tocar el resultado, ¿el nombre seguiría siendo verdad?

## Parte 2 — Renombra

Reescribe los cinco nombres con `Metodo_Escenario_ResultadoEsperado`, leyendo cada cuerpo para que el
titular cuente la noticia. Nombres en castellano (el método arranca con su nombre real del repo).

## Parte 3 — Lee la suite como especificación

Pon tus cinco nombres nuevos en una lista y léelos de corrido. ¿Suena a la especificación de lo que
hacen la calculadora y el pedido? Compara con los tests reales del repo, que ya siguen el formato:
`tests/VentasShop.TestsUnitarios/CalculadoraDescuentosTests.cs` y `PedidoEstadosTests.cs`.

---

## Solución comentada

**Parte 1.**
1. El más peligroso es el **4**, `Pagar_PedidoConfirmado_PasaAPagado_OtroMas`: su nombre dice que prueba
   el pago, pero el cuerpo prueba el envío. Un nombre que **miente** es peor que uno vago, porque te da
   información falsa justo cuando confías en ella al leer el panel en rojo. Los vagos (`Test1`,
   `TestDescuento2`, `TestExcepcion`) no te dicen nada, pero al menos no te engañan.
2. No. El nombre pasaría a mentir sin que el comportamiento haya cambiado, porque nombra el *cómo*
   (`UsaSwitch`) en vez del *qué*. Nombra siempre el comportamiento.

**Parte 2.** Una versión válida:

```csharp
// 1)
CalcularTasaDescuento_PedidoDe500ClienteEstandar_Aplica10Porciento
// 2)
CalcularTasaDescuento_VipConPedidoGrande_AplicaElTopeDel15
// 3)
CalcularTasaDescuento_PedidoMedio_Aplica5Porciento
// 4)  el nombre ahora dice lo que el cuerpo hace de verdad
Enviar_PedidoPagado_PasaAEnviado
// 5)
Pagar_PedidoSinLineas_LanzaPedidoSinLineas
```

**Parte 3.** Leídos de corrido suenan a la especificación: "calcula 10% en un pedido de 500 estándar;
aplica el tope del 15% al VIP con pedido grande; 5% en el pedido medio; un pedido pagado pasa a enviado;
pagar sin líneas lanza la excepción". Eso es documentación viva: si mañana el código deja de cumplir
cualquiera de esas frases, el test se pone rojo y te avisa. Si tu lista **no** suena a especificación,
te faltan nombres o alguno sigue mintiendo.

> Detalle: el test 2 mezcla dos reglas (volumen + tipo) que confluyen en el tope; el nombre se centra en
> el resultado observable (el tope del 15%), no en la mecánica interna. Nombra lo que el lector necesita
> saber cuando lo vea en rojo.
