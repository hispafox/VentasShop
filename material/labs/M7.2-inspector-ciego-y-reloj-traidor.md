# Lab M7.2 · El inspector ciego y el reloj traidor

> Submódulo M7.2. Tiempo: ~45-55 min. Dos ejercicios, uno por enfermedad: cazar la aserción débil con
> mutation testing, y curar el test inestable que depende del reloj. La entrega no es un número más alto:
> es una suite donde cada verde significa algo.

## Ejercicio 1 — El inspector ciego (mutation testing)

Punto de partida: [`AsercionesDebilesTests`](../../tests/VentasShop.TestsUnitarios/AsercionesDebilesTests.cs),
con tres aserciones débiles a propósito sobre la `CalculadoraDescuentos` y una fuerte de contraste.

1. Instala Stryker y pásalo (config en `stryker-config.json`, muta `CalculadoraDescuentos` y `Cupon`):

   ```bash
   dotnet tool install -g dotnet-stryker
   cd tests/VentasShop.TestsUnitarios
   dotnet stryker --test-runner mtp
   ```

   **Por qué `--test-runner mtp`:** el proyecto va sobre MTP (sin VSTest); sin la bandera, Stryker no
   encuentra los tests (en *preview* desde Stryker 4.13).

2. Abre `StrykerOutput/.../reports/mutation-report.html`. Verás el mutation score (~87,5% en esta rama) y
   **un mutante superviviente** sobre la `CalculadoraDescuentos`: el que cambia el tramo de volumen a cero.
   Ningún test débil lo mata, porque ninguno afirma el valor exacto.

3. Localiza el test culpable y dale la aserción que le falta (`Assert.Equal(0.15m, tasa)`). Relanza Stryker:
   ese mutante ahora **muere** y el score sube. No por escribir más tests: por hacer que los que ya tenías
   comprueben de verdad.

## Ejercicio 2 — El reloj traidor (flaky)

Punto de partida: [`RelojTraidorTests`](../../tests/VentasShop.TestsUnitarios/RelojTraidorTests.cs).

1. Mira el test marcado `[Fact(Skip=...)]`: usa `DateTimeOffset.Now` para validar un `Cupon` que caduca el
   30-jun-2026. Pasa hoy, pero el día que la máquina tenga una fecha posterior se pondrá rojo sin que nadie
   toque el código. Es una bomba de relojería; por eso está `Skip` (si no, pudriría la suite).

2. Cúralo: en vez del reloj del sistema, fija «ahora» con `RelojFijo` (el stub de IReloj de M5.1). Tienes el
   patrón en el test determinista de al lado: compruebas el cupón válido **antes** de expirar y caducado
   **después**, las dos con fecha fija. El test pasa siempre o falla siempre, por una razón.

3. Quítale el `Skip` a tu versión curada y comprueba que la suite sigue verde, sea cual sea el día.

## Comprobación

```bash
dotnet test tests/VentasShop.TestsUnitarios     # 78 verdes + 1 skip (el frágil)
```

## Lo que viene

M7.3 cierra el módulo con la higiene que evita de raíz las aserciones débiles, la inestabilidad y la
fragilidad: un assert conceptual por test, independencia, simplicidad y buenas prácticas.
