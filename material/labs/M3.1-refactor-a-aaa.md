# Lab M3.1 — Refactorizar a Arrange-Act-Assert

> En este lab no añades comportamiento: coges tests mal estructurados y los dejas legibles. Sin cambiar
> **qué** prueban, solo **cómo** lo prueban. El objetivo es que el test te señale el sitio con el dedo
> cuando algo se rompa.

## El punto de partida

Imagina que te encuentras estos dos tests en la suite. Los dos pasan en verde. Y los dos están mal.

```csharp
// ❌ Test 1: ¿qué prueba esto, exactamente?
[Fact]
public void TestPedido()
{
    var pedido = new Pedido(new Cliente { Nombre = "Ana", Tipo = TipoCliente.Vip });
    pedido.AgregarLinea(
        new Producto { Nombre = "Teclado", PrecioUnitario = new Dinero(50m, "EUR") },
        new Cantidad(2));
    Assert.Equal(EstadoPedido.Borrador, pedido.Estado);
    pedido.Confirmar();
    Assert.Equal(EstadoPedido.Confirmado, pedido.Estado);
    pedido.Pagar();
    Assert.Equal(EstadoPedido.Pagado, pedido.Estado);
    pedido.Enviar();
    Assert.Equal(EstadoPedido.Enviado, pedido.Estado);
}

// ❌ Test 2: el resultado esperado se calcula con la fórmula del código
[Fact]
public void TestDescuento()
{
    decimal total = 500m;
    var calc = new CalculadoraDescuentos();
    decimal tasa = calc.CalcularTasaDescuento(total, TipoCliente.Estandar);
    Assert.Equal(total >= 500m ? 0.10m : 0.05m, tasa);   // misma lógica que el SUT
}
```

## Parte 1 — Diagnostica

Antes de tocar nada, responde:

1. El **Test 1** mezcla cuatro comportamientos. ¿Qué te dice el reporte si falla la comprobación de
   después de `Pagar()`? ¿Qué pasa con `Enviar()`?
2. El **Test 2** lleva una aserción, pero hay trampa. Si el `CalcularTasaDescuento` tuviera mal el tramo
   de los 500, ¿lo cazaría este test? ¿Por qué no?

## Parte 2 — Refactoriza

1. Parte el **Test 1** en tests AAA, uno por comportamiento, con nombre que hable
   (`Pagar_PedidoConfirmado_PasaAPagado`, etc.). Fíjate en que el ciclo ya está partido así en
   `tests/VentasShop.TestsUnitarios/PedidoEstadosTests.cs`: úsalo de modelo.
2. Arregla el **Test 2**: el valor esperado se escribe **a mano**, literal (`0.10m`), no se calcula con
   la fórmula del SUT. Separa las tres fases con sus comentarios `// Arrange`, `// Act`, `// Assert`.
3. Compara tu resultado con `tests/VentasShop.TestsUnitarios/EstructuraAaaTests.cs`, que tiene el AAA
   canónico y el caso de "un concepto, dos asserts" (`CalcularTotal`).

## Parte 3 — Compruébalo

```bash
dotnet test tests/VentasShop.TestsUnitarios
```

Rompe a propósito una transición del `Pedido` (por ejemplo, que `Pagar()` deje el estado en
`Confirmado`), ejecuta, y mira **qué** test se pone rojo. Con el `TestPedido` de cuatro-en-uno solo
sabrías que "algo del pedido" falló; con los tests partidos, el reporte te nombra el comportamiento
exacto. Deshaz el cambio después.

---

## Solución comentada

**Parte 1.**
1. El reporte solo diría "TestPedido falló": no sabrías si reventó al confirmar, al pagar o al enviar. Y
   como `Pagar()` lanzaría antes, la comprobación de `Enviar()` **ni se ejecuta**. Pierdes información
   justo cuando más la necesitas. Un comportamiento, un test.
2. No lo cazaría. `total >= 500m ? 0.10m : 0.05m` es **la misma lógica que el SUT**: si el código está
   mal en la frontera, el test estará mal igual, y los dos errores se tapan. El test comprueba que la
   fórmula es igual a sí misma, no que el resultado sea el correcto. Por eso el esperado va literal.

**Parte 2.** El Test 1 se convierte en varios tests como los de `PedidoEstadosTests` (cada transición,
la suya). El Test 2 queda así:

```csharp
[Fact]
public void CalcularTasaDescuento_PedidoDe500ClienteEstandar_Aplica10Porciento()
{
    // Arrange
    var calculadora = new CalculadoraDescuentos();
    decimal totalPedido = 500m;
    var tipo = TipoCliente.Estandar;

    // Act
    decimal tasa = calculadora.CalcularTasaDescuento(totalPedido, tipo);

    // Assert
    Assert.Equal(0.10m, tasa);   // valor literal, verificación independiente
}
```

**Parte 3.** Con los tests partidos, romper `Pagar()` pone en rojo exactamente
`Pagar_PedidoConfirmado_PasaAPagado` (y deja en verde los demás). Esa es la diferencia: el test
estructurado en AAA te señala el sitio con el dedo. La estructura no es estética; es lo que hace que la
red de seguridad siga sirviendo dentro de dos años.

> El `Arrange` del Test 1 era ruidoso (crear cliente, producto, línea…). Esconderlo en una línea que se
> lee —un Builder— es justo lo que viene en M3.3.
