# Lab M7.3 · De suite-jungla a suite-jardín

> Submódulo M7.3. Tiempo: ~45-55 min. No escribes tests nuevos: **cuidas** los que hay. Coges una suite
> descuidada y la dejas como un jardín, aplicando los hábitos del capítulo uno por uno. La entrega no es más
> cobertura: es la misma protección con la mitad del ruido.

## El punto de partida: la suite-jungla (el «antes»)

Imagina estos tests descuidados sobre VentasShop (los hábitos rotos están marcados):

```csharp
// ❌ 1. Lo comprueba todo: 4 comportamientos, nombre mudo, la 1.ª aserción que falla corta el resto.
[Fact]
public void TestPagar2()
{
    var pedido = new PedidoBuilder().ConLinea(50m, 2).Confirmado().Build();
    pedido.Pagar();
    Assert.Equal(EstadoPedido.Pagado, pedido.Estado);
    Assert.Equal(100m, pedido.CalcularTotal().Importe);
    Assert.Single(pedido.Lineas);
    Assert.Equal(TipoCliente.Estandar, pedido.Cliente.Tipo);
}

// ❌ 2. Lógica en el test: un for recorriendo casos. Si uno falla, no sabes cuál.
[Fact]
public void TestDescuentos()
{
    var calc = new CalculadoraDescuentos();
    var casos = new[] { (500m, 0.10m), (100m, 0.05m) };
    foreach (var (total, esperado) in casos)
        Assert.Equal(esperado, calc.CalcularTasaDescuento(total, TipoCliente.Estandar));
}

// ❌ 3. Estado compartido: dos tests con una variable estática; fallan según el orden.
private static int _contador;
[Fact] public void TestA() { _contador++; Assert.Equal(1, _contador); }
[Fact] public void TestB() { _contador++; Assert.Equal(1, _contador); }

// ❌ 4. Test de una funcionalidad eliminada (p. ej. un "modo regalo" que ya no existe): mala hierba.
```

## El destino: la suite-jardín (el «después»)

Está en [`tests/.../BuenasPracticasTests.cs`](../../tests/VentasShop.TestsUnitarios/BuenasPracticasTests.cs).
Tu trabajo es transformar la jungla en eso:

1. **Separa el `TestPagar2`** en tests de un comportamiento cada uno, con su nombre que canta
   (`Pagar_DePedidoConfirmado_DejaElPedidoEnPagado`, `CalcularTotal_DeDosUnidadesA50_Devuelve100`). Un assert
   conceptual por test.
2. **Convierte el `for` en una `[Theory]`** con `[InlineData]`: cada caso, una ejecución independiente, sin
   lógica tuya. (Ojo al truco de M4.2 con decimales: pasa la tasa esperada como `double` y castea, o usa
   `TheoryData<decimal,...>`.)
3. **Rompe la dependencia de orden**: fuera la variable estática `_contador`; cada test fabrica su propio
   mundo con el `PedidoBuilder`. Ahora el orden de ejecución da igual (FIRST, M1.3).
4. **Renombra** lo que no dice nada (`TestPagar2` → un nombre que documente el comportamiento).
5. **Borra sin culpa** el test de la funcionalidad muerta: no protege nada y da una imagen falsa de lo que
   el sistema hace. Git se acuerda si algún día lo necesitas.

## Comprobación

```bash
dotnet test tests/VentasShop.TestsUnitarios     # verde
```

Compara el antes y el después: el mismo número de comportamientos cubiertos, pero una suite que se lee, en la
que un rojo te dice al instante qué se rompió, y a la que da gusto añadir el siguiente test.
