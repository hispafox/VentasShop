# Lab M3.3 — Object Mother y Test Data Builder

> En este lab construyes el atrezzo de test de VentasShop y reescribes con él un Arrange enredado. El
> objetivo: que a la vista del test quede solo la condición que prueba, y que el andamiaje se esconda.

## El punto de partida

En la rama tienes `tests/VentasShop.TestsUnitarios/PedidoEstadosTests.cs`, que monta el pedido a mano con
un helper privado:

```csharp
private static Pedido PedidoConfirmado()
{
    var cliente = new Cliente { Nombre = "Cliente de prueba", Tipo = TipoCliente.Estandar };
    var producto = new Producto { Nombre = "Producto de prueba", PrecioUnitario = new Dinero(50m, "EUR") };
    var pedido = new Pedido(cliente);
    pedido.AgregarLinea(producto, new Cantidad(2));
    pedido.Confirmar();
    return pedido;
}
```

Funciona, pero ese montaje vive solo en esa clase. En cuanto otro test necesite un pedido confirmado,
copias el bloque. Vamos a sacarlo a atrezzo reutilizable.

## Parte 1 — El Object Mother

Mira `tests/VentasShop.TestsUnitarios/Mothers/ClienteMother.cs`. Tiene tres arquetipos
(`Estandar()`, `Premium()`, `Vip()`). Añade tú un cuarto método que te haga falta y comprueba que el
test lo lee de un vistazo:

```csharp
var cliente = ClienteMother.Vip();   // "dame un cliente VIP"; los demás campos dan igual
```

¿Por qué un Object Mother y no un `new Cliente { ... }` en cada test? Responde en una línea.

## Parte 2 — El Builder

Mira `tests/VentasShop.TestsUnitarios/Builders/PedidoBuilder.cs`. Es el taller: encadena decisiones y
construye al llamar `Build()`. Cada método devuelve `this` (por eso encadena) y rellena con valores por
defecto razonables lo que no especifiques.

1. Escribe un test que use `new PedidoBuilder().ParaVip().ConLinea(300m, 2).Pagado().Build()` y comprueba
   el total. Léelo en alto: ¿se entiende el escenario sin abrir el Builder?
2. Escribe el caso de "pagar sin líneas" con `new PedidoBuilder().SinLineas().Build()`. ¿Por qué
   `SinLineas()` en estado borrador **no** inyecta la línea por defecto, y `Confirmado()` **sí**?

## Parte 3 — El Mother que usa el Builder

Mira `Mothers/PedidoMother.cs`: cada arquetipo (`Confirmado()`, `Pagado()`, `VipPagado()`) es, por
dentro, un `PedidoBuilder` configurado. Es lo mejor de los dos patrones. Añade el arquetipo que te
parezca que la suite va a repetir más, y úsalo en un test.

## Parte 4 — Reescribe

Coge un test con Arrange a mano (el helper `PedidoConfirmado()` de `PedidoEstadosTests`, por ejemplo) y
reescríbelo con el Builder/Mother. Compara las dos versiones: ¿cuál se lee mejor? ¿Cuántos sitios tocas
si mañana cambia el constructor de `Pedido`?

```bash
dotnet test tests/VentasShop.TestsUnitarios
```

---

## Solución comentada

**Parte 1.** Un `new Cliente { ... }` en cada test repite el montaje y es un sitio más que tocar cuando
cambie el constructor. El Object Mother centraliza el atrezzo: un punto único de cambio, y el test se
lee como una intención ("un VIP") en vez de como una construcción.

**Parte 2.**
1. Ese test ya está en `ConstruccionDatosTests.cs` (`CalcularTotal_PedidoVipConDosUnidadesDe300_Da600Euros`,
   vía `PedidoMother.VipPagado()`). El escenario se lee en la propia línea.
2. Porque el Builder solo inyecta la línea por defecto cuando pides un **estado avanzado**
   (`Confirmado`/`Pagado`), que por regla de negocio (BR-07) exige al menos una línea. En estado
   `Borrador`, `SinLineas()` deja el pedido vacío **a propósito**: es justo lo que necesita el test de
   "pagar sin líneas lanza la excepción". El atrezzo tiene defaults razonables, pero no te quita el caso
   límite cuando lo pides explícitamente.

**Parte 3.** Un `PedidoMother` que por dentro usa el `PedidoBuilder` te da la comodidad del Mother para
lo habitual y la flexibilidad del Builder por debajo. `ClienteMother.Vip()` podría ser, igual,
`new ClienteBuilder().Vip().Build()` el día que el cliente necesite combinatoria.

**Parte 4.** La versión con Builder/Mother se lee mejor (el Arrange es una frase) y, sobre todo, un
cambio en el constructor de `Pedido` se arregla en **un** sitio (el Builder) en vez de en cada test.
Ese es el valor que se amortiza a partir del quinto test que reutiliza el atrezzo. Y ojo: el atrezzo es
una inversión, no se construye por adelantado — el primer test, a mano; cuando ves el patrón repetirse,
lo sacas.

> Estos Builders y Mothers se quedan como **plantillas reutilizables** del repo y los usaremos sin
> volver a explicarlos en los Módulos 4, 5 y 6.
