# Lab M6.3 · El contraste: SQLite in-memory contra el simulador

> Submódulo M6.3. Tiempo: ~35-45 min. Coges los tests de integración de M6.2 y los ejecutas también contra
> **SQLite in-memory**, un motor relacional de verdad. El de la unicidad, que en M6.2 dejaba pasar, ahora
> salta. Sin instalar ni levantar nada.

## Lo que vas a conseguir

Montar tests contra un motor relacional real en RAM, ver saltar la restricción de unicidad que el provider
in-memory no cazaba, y entender el detalle que más despista: la base vive mientras la conexión esté abierta.
Es el coche de verdad en el circuito de pruebas, después del simulador de M6.2.

## Punto de partida

El modelo resuelto está en
[`tests/.../RepositorioPedidosSqliteTests.cs`](../../tests/VentasShop.TestsIntegracion/RepositorioPedidosSqliteTests.cs).
`Microsoft.EntityFrameworkCore.Sqlite` ya está en el `.csproj` (junto al provider in-memory de M6.2, que se
conserva para el contraste). `Producto` ya trae el campo `Codigo` con índice único en
[`ContextoVentasShop`](../../src/VentasShop.Infraestructura/Persistencia/ContextoVentasShop.cs).

## Parte 1 — La conexión que se mantiene abierta

Monta el fixture por test con `IDisposable`. La clave: abres la conexión en el constructor y la mantienes
viva; a `UseSqlite` le pasas el **objeto** conexión, no una cadena.

```csharp
public class RepositorioPedidosSqliteTests : IDisposable
{
    private readonly SqliteConnection _conexion;

    public RepositorioPedidosSqliteTests()
    {
        _conexion = new SqliteConnection("Filename=:memory:");
        _conexion.Open();
        using var contexto = NuevoContexto();
        contexto.Database.EnsureCreated();   // crea el esquema (con el índice único)
    }

    private ContextoVentasShop NuevoContexto() =>
        new(new DbContextOptionsBuilder<ContextoVentasShop>().UseSqlite(_conexion).Options);

    public void Dispose() => _conexion.Dispose();   // cierra → se borra la base
}
```

Como xUnit crea una instancia de la clase por test, cada uno arranca con su base limpia y propia.

## Parte 2 — Guardar y recuperar (igual que en M6.2)

Reescribe el test de guardar un pedido y recuperarlo con sus líneas. Usa `NuevoContexto()` para guardar y
**otro** `NuevoContexto()` (misma conexión, misma base) para leer, con `Include(p => p.Lineas)`. Comprueba
que pasa igual que contra el provider in-memory: la lógica de consulta se comporta igual en los dos.

## Parte 3 — La recompensa: la unicidad ahora SÍ salta

Copia el escenario de la unicidad de M6.2 (dos productos con el mismo `Codigo`) y, esta vez, **espera la
excepción**:

```csharp
contexto.Productos.Add(ProductoCon("ABC", 50m));
contexto.Productos.Add(ProductoCon("ABC", 20m));

Action guardar = () => contexto.SaveChanges();
guardar.Should().Throw<DbUpdateException>();   // contra SQLite SÍ salta
```

Ponlos al lado del de M6.2 (`...Should().BeNull()`) y léelos juntos: el simulador y el coche de verdad, en
una sola suite.

## Parte 4 — Provoca el fallo del módulo (para reconocerlo)

A propósito, **no abras** la conexión (o ciérrala antes de tiempo) y ejecuta un test. Verás el error de
«tablas que no existen»: es la base que ha desaparecido. Es el error nº1 con SQLite in-memory; provocarlo una
vez es la mejor forma de no caer en él en producción.

## El límite que no olvidas

SQLite es un motor relacional real, pero no es tu motor de producción: hay diferencias de dialecto. Cubre el
grueso de la integración sin montar nada; para el dialecto exacto crítico, el juez es tu motor real.
