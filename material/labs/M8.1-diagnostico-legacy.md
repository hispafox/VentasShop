# Lab M8.1 · Diagnóstico de un legacy y refactor mínimo

> Submódulo M8.1. Tiempo: ~45-55 min. Haces de médico de guardia con un legacy neutro: triaje, diagnóstico
> de testabilidad (los clavos) e intervención mínima (abrir las costuras con DI, sin tocar la lógica). La
> entrega no es una suite de tests: es un proyecto que antes no se podía testear y ahora sí.

## El «antes»: el legacy con sus clavos

Este es el `NotificadorPedidos` tal como llega (neutro, inventado). **No compila en un test ni se deja
testear**, y ese es el punto:

```csharp
public class NotificadorPedidos
{
    public void NotificarEnvio(Guid pedidoId)
    {
        var repo = new RepositorioPedidosSql("Server=prod-db;Database=Ventas;...");  // clavo 1
        var pedido = repo.Cargar(pedidoId);

        if (DateTime.Now.Hour < 22)                          // clavo 2: reloj del sistema
        {
            var smtp = new SmtpClient("smtp.empresa.local");
            smtp.Send(pedido.ClienteEmail, "Tu pedido va de camino");   // clavo 3: correo real
        }

        Logger.Instance.Log($"Pedido {pedidoId} notificado");           // clavo 4: singleton estático
    }
}
```

Intenta escribirle un test unitario. No puedes: conectaría a una base real, dependería de la hora, mandaría
un correo de verdad y llamaría a un global que no puedes sustituir.

## Parte 1 — Triaje y diagnóstico

1. Si tuvieras varias clases legacy, prioriza por riesgo × consecuencia (M2.1) y por el historial de Git.
2. En esta clase, **señala los 4 clavos** y di, para cada uno, por qué impide el test. Son las dependencias
   que hay que convertir en **costuras**.

## Parte 2 — La intervención mínima (abrir las costuras)

Convierte cada clavo en un parámetro del constructor (inyección de dependencias), **sin cambiar la lógica**.
El resultado está resuelto en
[`tests/.../Legacy/NotificadorPedidos.cs`](../../tests/VentasShop.TestsUnitarios/Legacy/NotificadorPedidos.cs):
`IAccesoPedidos`, `IReloj`, `IEnviadorCorreo`, `ILogger<>`. La lógica de dentro (cargar, mirar la hora, enviar
si es de día, registrar) es idéntica línea a línea: **cambias la estructura, no el comportamiento**.

Apóyate en el refactor del IDE (extraer interfaz, introducir parámetro): son cambios mecánicos y de bajísimo
riesgo, lo bastante seguros como para hacerlos *antes* de tener tests. Si algún cambio te diera respeto,
pondrías primero un **test de caracterización** que fije el comportamiento actual.

## Parte 3 — La recompensa

Con las costuras abiertas, la clase ya se testea. En
[`Legacy/NotificadorPedidosTests.cs`](../../tests/VentasShop.TestsUnitarios/Legacy/NotificadorPedidosTests.cs)
verás los dobles en acción (RelojFijo y LoggerEspia de M5.1 + un acceso falso y un enviador espía): notifica
de día, no de noche, y deja apunte en el log.
